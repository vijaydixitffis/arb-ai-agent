Pre-ARB AI Agent — Domain Table Schemas

LLM-populated structured records per architecture domain · Orchestrator envelope + 9 domain agents
v1.0
Each domain agent calls the LLM API once with its retrieved knowledge-base context + SA artifacts, and returns a structured JSON object conforming to the DomainReviewPayload schema. The orchestrator merges all payloads into a single ReviewSession document and drives the HITL gate.

Table hierarchy
ReviewSession (root)Orchestrator
Top-level document. One per ARB review. Contains session metadata, aggregate score, recommended decision, and an array of DomainReviewPayload objects (one per agent) plus the NFR scorecard array and cross-platform impact records.
DomainReviewPayload (per agent)Each domain agent
One object per domain. Contains: Summary (1 record) · Blockers[] · Recommendations[] · Findings[] · Actions[] · ADRs[]. Each table is an array that the LLM populates in a single structured output call.

One DomainSummary record per domain per review. The LLM generates this after completing all findings and scoring, so it can reference confirmed scores and blockers. It is the single source of truth for the domain's headline status.

Recommendations are proactive, forward-looking and distinct from Findings (which are reactive). Each recommendation must be actionable and reference a specific approved pattern or standard. The LLM must generate at least one recommendation per domain, even for GREEN domains.

Actions are the post-ARB accountability mechanism. Every BLOCKER and every AMBER finding must have at least one action. Actions generated by the agent are provisional — EA may adjust owner, due date or verification method at Gate 2. The consolidated action register is the primary monitoring surface post-ARB.

Each domain agent constructs a single LLM API call using this prompt template. The LLM returns the entire DomainReviewPayload as a JSON object in one call using response_format: json_object. This avoids multi-turn overhead and makes retries simple.
System prompt structure
// System prompt sent by every domain agent (domain-specific values injected at {} SYSTEM: You are the {DOMAIN_LABEL} specialist agent in the Pre-ARB AI Agent pipeline. Your role is to validate the Solution Architect's submitted artifacts against enterprise architecture standards and produce a structured JSON review payload. RULES: 1. Respond ONLY with a valid JSON object matching DomainReviewPayload schema. No preamble, no markdown, no explanation outside the JSON. 2. Every finding must reference a specific SA artifact AND a specific KB document. Never generate findings from general knowledge alone. 3. Every Blocker must have severity = "BLOCKER" and rag_score = 1. 4. Every AMBER or RED finding must have at least one corresponding Action. 5. Every significant decision must have a corresponding ADR. 6. Waivers (ADR type = WAIVER) must include a proposed waiver_expiry_date. 7. The domain rag_score must equal min(all finding rag_scores). You cannot score AMBER if any finding is RED. 8. If evidence is absent for a mandatory check, rag_score = 1 (RED). Never skip. 9. Do not invent evidence. If you cannot find it in the provided artifacts, flag it. 10. Security domain: any rag_score < 4 must generate a Blocker record. SCORING RULES: 5 = Fully compliant, all evidence present, no gaps 4 = Compliant with minor tracked actions only 3 = Partially compliant, gaps with mitigation plan 2 = Significant gaps, mandatory actions before go-live 1 = Fails mandatory standard or evidence absent (RED / BLOCKER)

Orchestrator aggregation logic
1
Await all domain agent results
Orchestrator fans out 9 parallel LLM calls. Waits for all to complete (or timeout after SLA). Marks failed agents with status AGENT_ERROR and logs for retry.
2
Flatten blockers and actions
Iterates domain_reviews[], collects all Blocker[] and Action[] into root consolidated_blockers[] and consolidated_actions[]. Assigns global sequential IDs cross-domain.
3
Compute aggregate_rag_score
Rule: min(all domain summary rag_scores). Override: if any is_security_or_dr blocker exists → aggregate forced to RED (1) regardless of other scores.
4
Determine recommended_decision
Score 5 → APPROVE. Score 4 → APPROVE (minor actions). Score 3–2 → APPROVE_WITH_CONDITIONS. Score 1 with security/DR blocker → REJECT. Score 1 other → DEFER.
5
Generate decision_rationale (LLM call)
Orchestrator makes one additional LLM call with the aggregated scores and consolidated blockers to write the 2–4 sentence decision rationale. This is the only orchestrator LLM call.
6
Trigger HITL Gate 2
Sets status = EA_REVIEW, writes ReviewSession to store, sends EA notification via ServiceNow/Teams with dossier link. SLA timer starts for EA review (1 business day).
